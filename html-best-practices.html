HTML Best Practices

Semantics

Using semantic HTML is important for accessibility, as it allows assistive technology to intelligently traverse the page contents. Semantic HTML can also help with search engine optimization. Be sure to use the proper element for the content.

  • Use standard elements such as p and hx for paragraphs and headers.
  • Use HTML5 tags like article, section, etc. properly to create a sensible hierarchy. Use this flowchart as a way to determine how your content should be tagged: HTML5 tag decision tree.

Naming Conventions

Don't abbreviate words in class names unless there's a constraint with an existing name/element/function/et al. It's bad for SEO and for developer comprehension.

When naming files that will appear as part of a URL, it is imperative to use a hyphen (-) between words. Search engines 'read' a hyphen as a space, while they interpret underscores as an actual character, thus skewing the readability of the keyword(s). Camel case is also an inaccurate way to make filename-related keywords readable by a search engine. They are fine for classes used for CSS or JS hooks as a developer aid, but they should not be used when naming any HTML/JSP file.

Syntax

  • Always use double quotes around HTML element attributes.
  • Do not leave attribute values empty unless it is required due to programmatic reasons.
  • IDs can be used in markup, however, in a few certain situations. The best example is for anchor links within a page. They can also be used as Javscript hooks where a single element is targeted; again, prefixing the name with "js-" is a good practice. This article goes over some of these exceptions. In general, they are not useful in writing CSS because they are not resuable, which is against the nature of Cascading Style Sheets.

Formatting

  • Set tabs to 4 spaces in your code editor (also known as Soft Tabs). This way, everyone can set their own viewing preference while keeping the code normalized at one tab character.
  • Be cognizant of removing extra spaces, trailing whitespace, extra empty lines, etc, but don't get bogged down in it. We will rectify any extra spacing or empty lines on the server side once the code is rendered to the screen.
  • Never leave any space before your doctype declaration at the top of the page. This is the first step to ensuring Internet Explorer doesn't go into Quirks mode.
  • Try to keep HTML comments out of the main section of code. Comments at the beginning or end of a file make commenting out blocks of HTML much easier than comments dispersed throughout the code.
  • Use comments to designate a closing tag: 
    <div id="id-name">
...
</div><!-- /#id-name -->
<span class="class-name">
...
</span><!-- /.class-name -->

Testing

  • Cross-browser test your code before submitting your file for code review.
  • Run your code through a W3C-style validator before submitting your file for code review.

Caution

  • Be cautious when using ARIA attributes which require mapping to an ID. This can cause programmatic confusion as we should be avoiding the ID attribute to remain clear of programmatic territory.
  • Be cautious when using web fonts using the @font-face technology. Among other reasons, there are these to consider.

Nevers

  • Never use "@import" to reference an external stylesheet
  • Never use inline styles
  • Never use uppercase for element tags
  • Never use tables for layout
