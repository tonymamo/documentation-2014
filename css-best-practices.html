CSS Best Practices

Naming Conventions

  • CSS class names should be sensible to other developers. Class names should communicate useful information to fellow developers. They help to provide hooks for the presentational or behavioral layers to semantic HTML.
  • Avoid systematic use of abbreviated class names. Don't make things difficult to understand. Leave plenty of comments if there could possibly be any confusion for a different developer at a later time.
  • The most reusable components are those with class names that are independent of the content. Avoid contextual information in a class name such as a color or location (i.e. .left-sidebar).
  • Class names should be lowercase and should only use hyphens: class="horizontal-list". Class names should not use camelCase or under_scores.
  • Prefix Javascript hooks with "js-": js-submit. Don't add presentational styles to them, only use for behaviors. If something's style changes, append additional classes that can be used without JS so a non-JS environment such as Java can use same class(es) for progressive enhancement/graceful degradation.
  • Selectors for components should uses class names; avoid the use of generic tags.
  • Important Don't use #ids in your CSS! They add an order of magnitude that makes them difficult to override, usually leading to over-qualifying selectors and adding unnecessary classes or IDs elsewhere, like on the body. Refer to this article about writing efficient selectors by Harry Roberts of CSSwizardry.com to learn more about specificity in selectors.

Syntax

  • Always close the last declaration in a ruleset with a semicolon (;). It might be the last one now, but might not be later.
  • Leave a space before the opening brace and the colon in rulesets. It makes for quicker highlighting of text.
  • Close the ruleset with a brace in the same column as the rule name
    code example 
    .class-name {
	property: value;
}
  • Line up vendor prefixes so their properties line up, it can speed up changing values, especially in editors that allow multi-line column select
    code example 
    .sample-class {
  -webkit-box-shadow: 1px 10px 5px #458;
     -moz-box-shadow: 1px 10px 5px #458;
	      box-shadow: 1px 10px 5px #458;
}
  • Don't overqualify selectors. .nav{} is faster and more robust/reusable than ul.nav{}. See this article on efficient css selectors and this article about selector intent explaining this idea.
  • Use one discrete selector per line in multi-selector rulesets
    code example 
    .selector,              
.selector-name1,        
.selector-name2 {       
	display: block;     
    float: none;        
    background: url("x.png"); 
    margin: 0;          
}
  • Preferably, always use double quotes, in case there are contractions or something quoted inside of an attribute. Also useful for background: url(); and content: " "; which require quotes.
  • Where applicable, avoid specifying units for zero-values.
  • When possible, write your CSS properties in shorthand. Instead of having margin-top, margin-bottom, margin-left, and margin-right, just declare margin once and combine their values. Keep in mind that the order goes clockwise starting at the top. You can remember this with the mnemomic device TRBL (trouble), which stands for Top, Right, Bottom, Left.
    code example 
    /* Common Shorthand Declarations */
.element {
    margin: top | right | bottom | left; /* explicit */
    margin: top/bottom | left/right; /* symmetrical shorthand */
    padding: top | right | bottom | left; /* explicit */
    padding: top/bottom | left/right; /* symmetrical shorthand */
    background: color | url | repeat | position | attachment;
    font: style | weight | size/line-height | family;
    border: width | style | color;
    list-style: type | position | image;
}
  • Writing in shorthand will prevent errors and others overriding values higher up in a long ruleset. If a padding-top is set, and then padding a few lines down, it will override the first declaration, even if you didn't mean to. By ordering properties according to the Ordering section above, and using shorthand, we can avoid these gotchas altogether.

Formatting

  • Always use the multi-line format for writing CSS, even if there is only one declaration. The main reason for this is for using Version Control like Subversion. It is much easier to do a per-line Diff/Compare than to try and find the difference in one long line. Also, multi-line helps in readability and scanability for developers.
    code example 
    /* GOOD */
.selector-name1,
.selector-name2 {
    display: block;
    color: #666;
    background: #bada55;
}           

    /* BAD */
.selector-name1, .selector-name2 { display: block; color: #666; background: #bada55;}
  • We are not concerned about file size in the authored, formatted CSS file as it will be compiled and minified on deployment. Let's use this to our advantage and make our stylesheets comprehensible and organized for maintainability.
  • Use only one discrete selector per line, and indent the ruleset once. There should also be only one rule per line as well. This will make for a very consistent stylesheet and save headaches down the road.
  • The order in which you write properties actually does not matter, but writing it in a consistent way can help one understand how one thing will affect another. The general preference is for structurally important properties (e.g. positioning and box-model) to be declared prior to all others, and to group things that belong together.
    code example 
    .element {
    /* BOX */
    display: ;
    visibility: ;
    float: ;
    clear: ;
    position: ;
    top: ;
    right: ;
    bottom: ;
    left: ;
    z-index: ;
    width: ;
    height: ;
    overflow: ;
    margin: ;
    padding: ;
    /* BORDER */	
    border: ;
    /* TEXT */
    font: ;
    text-align: ;
    text-transform: ;
    text-decoration: ;
    letter-spacing: ;
    word-spacing: ;
    white-space: ;
    vertical-align: ;
    color: ;
    /* BACKGROUND */
    background: ;
    /* OTHER */	
    list-style: ;
    table-layout: ;
    caption-side: ;
    border-collapse: ;
    border-spacing: ;
    empty-cells: ;
    opacity: ;
    cursor: ;
    content: ;
    quotes: ;
    /* VENDOR PREFIXES */
    -webkit- : ;
    -moz- : ;
    -ms- : ;
    -o- : ;
    unprefixed: ;
}
  • Set tabs to 4 spaces in your code editor (also known as Soft Tabs). This way, everyone can set their own viewing preference while keeping the code normalized at one tab character.
  • Separate each ruleset with a blank line
  • Make a Table of Contents (TOC) at the top of the CSS file.
    code example 
    /*------------------------------------*\
        TABLE OF CONTENTS
\*------------------------------------*/
/*
$BASE
$TYPOGRAPHY
$COLORS
*/ 

...

/*------------------------------------*\
        $Base
\*------------------------------------*/
  • Use large comments for section titles from the TOC, and prefix title with a $ sign to use a search filter to jump to a section. Using the $ allows you to narrow your search instead of querying everything that may use that same name as a .class in a section.
  • Leave 5 carriage returns (a.k.a. hard returns) after a section for ease of scanning through stylesheets (sections discussed under Commenting).
  • Leave comments with example HTML markup in them to demonstrate structure.

Best Practices for CSS Selectors

Target .classes, not an #id

Classes are reusable. Classes can be stacked. Javascript bound to classes should be / are (by intent) reusable. Keep your code dry. id based styles can cause specificity wars.

There are a number of highly useful selectors that you might want to study

Don't make blanket statements (that's the CORE team / CSS reset's job). Generic content styles should be wrapped in a semantically named class (i.e., .content). Example LESS:

code example 
article {
    max-width: 45em;
    margin: 0 auto;
    
    .content {
        padding: 0 0 1em 0;
        border-bottom: 1px solid lightgrey;
        
        p, ul {
            line-height: 1.5;
            
            a {
            color: @fontColor;
            }
        }
    }
}

Reasons to use an id selector

None.

 

...ok, fine.

An id is reasonable only if you are cordoning off styles for a specific page / instance, and want to ensure you win the spec-war.

Note: using an id to ensure style override is preferable to the nuclear option.

Specificity Wars

"You give every id selector ("#whatever") a value of 100, every class selector (".whatever") a value of 10 and every HTML selector ("whatever") a value of 1. Then you add them all up and hey presto, you have the specificity value:

  • p has a specificity of 1 (1 HTML selector)
  • div p has a specificity of 2 (2 HTML selectors; 1+1)
  • .sith has a specificity of 10 (1 class selector)
  • div p.sith has a specificity of 12 (2 HTML selectors and a class selector; 1+1+10)
  • #sith has a specificity of 100 (1 id selector)
  • body #darkside .sith p has a specificity of 112 (HTML selector, id selector, class selector, HTML selector; 1+100+10+1)

If all of these examples were used, div p.sith (with a specificity of 12) would win out over div p (with a specificity of 2) and body #darkside .sith p would win out over all of them, regardless of the order." -- Andy Clarke, CSS: Specificity Wars
