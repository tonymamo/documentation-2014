Object-Oriented CSS (OOCSS)

What is OOCSS?

Object Oriented CSS is a new way of thinking about how we author CSS files to make them small, efficient, flexible, and maintainable. The objective of OOCSS is to recognize patterns within the design, and abstract that pattern. By doing this, we reuse bits of CSS and avoid repeating ourselves unnecessarily within a style sheet.

A classic example of this is the horizontal list. Between navigation, breadcrumbs, and menus, most sites tend to have at least one or two horizontal lists. Normally, you would create classes for each list, and set the CSS property of display: inline-block;, or float: left; for each list. Why repeat this in our CSS? What happens when we need another horizontal list? The issue compounds each time another version of a list is added. OOCSS (built on the DRY (Don't Repeat Yourself) principle), provides a method of eliminating unnecessary code.

How to Use OOCSS

Getting started with OOCSS doesn't require learning any new CSS or HTML, but rather learning to think differently in how one crafts their style sheets.

The first step is to look at a page, whether it is a wireframe, design comp, or existing web page, and recognize the patterns that are being shared amongst different elements. Perhaps there are two or three colors, a box with an image on the left and content on the right, a border around different boxed elements, multiple horizontal lists, and more. Once you recognize these patterns, it is easy to see how a property: value; pair would be used multiple times on different elements, when in effect you are just repeating yourself unnecessarily. CSS is meant to cascade and be reusable, and the OOCSS methodology embraces that notion to leverage it the way it was supposed to be.

Once we recognize these patterns, the next step is to abstract these property: value; pairs into their own class. This is called your abstraction, or sometimes referred to as a base object or module. Optimallly, an abstraction should be basic and free of design-related styling, so that it can be extended with appended classes. This makes for extremely reusable code that is much easier to maintain in the future. It also allows for quick and easy reskinning or theming of a page without affecting other parts of a page.

To make better sense of this, we can make an analogy of objects and Legos. Imagine the different Lego shapes and their variations, but imagine them without color for a moment. These are analogous to base objects, or abstractions. They can be assembled in different ways, but contain the basic building blocks for limitless creations. Now, reintroduce your colors and variations of those Legos, which are like the extensions, or appending classes, that can be used to change the entire look and feel of a project.

A practical example is how Twitter Bootstrap handles their button styling. They have a grey button called .btn as their base object, which sets the rudimentary styling like padding, a simple background, the borders, text-shadow, and a little bit more. Then, you can extend the look of that base object button by adding supplementary appended classes to .btn to change both color and size. For example, you can have <button class="btn btn-primary btn-large"> , where the last two classes extended the first class. The first class set the base styling, the second class changed the color scheme, and the third changed the size. By leaving the base object mostly design-agnostic, it was able to take on these extensions without having to overwrite, or even worse, over-qualify it with more specificity to change the entire look and feel of it.

Between the 6 different color extensions and 3 different size extensions available in Bootstrap, you end up with 28 different potential results (base color + 6 extensions × Base size + 3 extensions). This is only 10 classes, as opposed to 28 classes had they been all written out separately. Think about how much file size was saved by reusing code instead of repeating it for each of the 28 different rulesets. This multi-class pattern of a base abstraction plus extensions is where OOCSS really shines.

Breaking Good Habits

By using OOCSS, we need to shift our thinking around what was used to be considered "best practices" in CSS. As HTML and CSS have progressed throughout the years, we have adopted some ideas around what makes for good and bad code, but when is the last time we really stepped back and analyzed why or what makes something good or bad?

Lately, we considered good habits to be avoiding using a lot of classes, using the least amount of markup possible to acheive something, and handcrafting our CSS. Conversely, the bad habits were "classitis", unsemantic class names, and being too verbose with markup. It may take some time to change our mindset, but these methods have now been proven to actually be less efficient, and there is now proof to debunk these bad habits.

  1. Avoiding Extra Classes/Classitis

    Contrary to popular belief, being generous with your classing everything can result in less markup and smaller CSS filesize. It seems counterintuitive, but if done properly (using OOCSS), file size can dramatically decrease even though there are more classes declared. The difference comes in when you are applying the DRY approach, and only state property: value; pairs a minimal number of times. So now, instead of a value being repeated numerous times in a stylesheet, it is only written once, and shared via class names.

    A memorable statistic that came out of Nicole Sullivan's research is how Facebook repeated the hex code for "Facebook Blue" a total of 261 times, in stylesheets that were Megabytes in size. If done correctly using OOCSS, there is no reason that number could have been less than 10. They may need to define it as a background-color:; , color:; , and border:; , but really not much more than that if at all. By embracing the cascade and letting CSS do the heavy lifting, they could have easily reduced their file size by at least 200 lines of code that were just stating the same hex value. After Nicole Sullivan and her research partners recoded a sample homepage of Facebook using abstractions and extensions, they were able to reduce the file size of the markup by almost 50% and the CSS by around 20%. In the end, there were actually more class names than before, but there were no styles that were based on an element and its position in the markup.

    By not styling on an element, all specificity wars were eliminated, and the CSS became much more maintainable, flexible, extensible, and predictible, all while being much smaller in filesize.

  2. Unsemantic Class Naming

    When it comes to naming a class, it doesn't matter if it is semantic or not. Class names are primarily used for hooks in CSS and JS, and make sense only to us developers. Of course, our markup should be as semantic as possible, because it will get interpreted by not only other developers, but by search engines, accessibility devices, and more. When it comes to classes, it only makes sense to talk about them in terms of sensible and insensible. They must make sense to us as a developer, and not need an explanation of what they mean.

    Refer to CSS Best Practices for more on naming conventions. To summarize, it is best to name a class based on what it does, not what it looks like, where it is located, or other contextual-based names.

  3. Using IDs in CSS

    Using IDs in a stylesheet should be avoided at all costs. They may be used in markup to serve as a hook for Javascript, but there is definitely no need for them to be used as a styling hook. They have no advantage over using a class; anything that can be styled with an ID can be done with a class.

    The problem with using them in CSS is because they go against the very nature of CSS itself: they are not reusable, and they introduce a high level of specificity, not unlike using !important. It is pretty hard to qualify a selector to be stronger than an ID, which means you will just be introducing highly over-qualified and long selector chains that go all the way up the DOM tree just to be stronger than an ID. Instead, why not use a class, which could then be reused later on, and also make your stylesheet more maintainable and extensible? By using a class over an ID, you are making your own and everyone else's lives easier in the future by avoiding the dreaded "Specificity Wars".

Putting It All Together

The main goal of OOCSS is to achieve the highest code quality possible. Before, that meant ensuring correctness, reliability, reusability, performance, testability, and extensibility. Now, with OOCSS, those all still apply, but come naturally when writing CSS with optimization for change in mind.

That means, while crafting your CSS, you anticipate maintenance later on, whether by you or another developer. If you do this correctly, all of those others will fall in line.

Tips & Tricks

  • Write CSS that has a 1:Many relationship with HTML elements. This ensures flexibility and that you are keeping your code DRY.
  • Write rulesets in order of specificity. This means instead of the order they occur in the markup, or just tacking on lines of code at the end of the file, that you order them by inheritance strength. Ideally, the base objects/abstractions come first, then extensions, and then various other classes. Subsequent rulesets should only inherit, not undo, something from higher up in the stylesheet (with minor exceptions).
  • Avoid over-qualifying your selectors by putting an element before the class (e.g. ul.nav instead of just .nav). It increases specificity, limits reusability, and is less efficient. Unlike us, CSS is processed by browsers from right to left, so it will look for all .nav and then for one inside of a ul, which takes much longer than just .nav.
  • Avoid unnecessary nesting and chaining of selectors in the stylesheet. Again, this will increase specificity, and becomes less efficient because it takes longer for the browser to parse. Also, it becomes less portable because it depends on HTML structure. The goal is to make selectors independent of structure so they can be reused elsewhere.

Understanding the Layers of CSS

Andy Hume gave a presentation on maturing best practices in which he describes how CSS actually works in different layers. To create good OOCSS code, he explains how abstractions and extensions should be written in a way that only affects one layer at a time, and provides a few examples of how it works. The four layers are the HTML Document, Base, Module, and Layout. If done correctly, what this essentially does is separate content and presentation.

The Record Set Abstraction

Nicole Sullivan pioneered one of the most useful abstractions, and kickstarted the whole OOCSS approach with her work. After analyzing Facebook, she realized that the UI layer followed a pretty basic pattern with an image or icon on the left, followed by some text or other content on the right. By abstracting this pattern, she was able to write just a few lines of code that would allow it to be infinitely nestable, have multiple configurations, and it must clearfix. It should also be able to have different widths, have no styling associated besides layout, and be agnostic about the content inside it. By doing so, hundreds of lines of repeated code was eliminated, and saw a file size reduction of close to 50%.
